Field-test updates (2026-06-27):
  • Homescreen autoplay → OFF. Confirmed the home-feed autoplay was popping MusicPlayingTrigger on app-open. Disabling it removes half of Bug B (the autoplay edge); the launch-edge half in the YT macro remains.
  • "Buttons still bump 200↔error while nothing plays." This is Bug A, not a new fault: the broken 10-min belt + the recursion macro keep re-validating with the unexpanded {lv=map[…]} payload (and now disagree with the manually-set ON state), so the floating buttons flip 200→409→422 every tick. The passthrough/backfill assert (Wave 1) deletes this churn.
  • Slay flap hypothesis (to test): Slay the Spire likely sits in the recents/background stack — swiping up to pick another app briefly surfaces it, firing a spurious ApplicationLaunchedTrigger that counts against the distraction tally even though it was never played. The sustained-foreground guard (Fix #5) is the countermeasure.
  • Stuck on/off: the 10-minute "belt & suspenders" re-assert sends an unexpanded variable, not a real boolean → server rejects it → self-heal never fires.
  • 200 & 500 (409) at once: two macros (YT on app-launch vs YT_BG on playback) assert opposite states within the same second — the floating buttons show both verdicts because both are live.
  • "Turns on when I open, not when I play": YT keys off ApplicationLaunchedTrigger (foreground), not playback; autoplay on the home feed then fires YT_BG.
  • Spotify never enters music mode: there is no server path from phone Spotify telemetry to music mode — the macro only clears YT.
  • Slay the Spire enforced while closed: games run on raw launch/close edges with no belt at all; spurious "launched" events flap and fire the zap cascade.

2 · Where it breaks

Bug A — core "stuck" bug

The 10-min re-assert sends an unexpanded variable

YT_BG's interval belt and play-edge both use {lv=map[{trigger_that_fired}]}. The local dict map only has keys for the two MusicPlayingTrigger names. When the interval trigger fires, {trigger_that_fired} = "Interval: 00:10:00", which is not a key → the lookup returns the literal string.
GET /api/state/validate?app=youtube&assert={lv=map[Interval: 00:10:00]} → 409 POST /phone/event {"play":"{lv=map[Interval: 00:10:00]}"} → 422
Effect: every 10 min the self-heal posts garbage the server rejects. Once state lands wrong, nothing restores it until a real play/stop edge. This is why it is stuck OFF right now.
Bug B — the "200 & 500" twist

YT (launch) and YT_BG (playback) assert opposite states

YT fires on ApplicationLaunchedTrigger and posts play:False; YT_BG fires on MusicPlayingTrigger and posts play:true (autoplay). On app-open both fire in the same second → contradictory edges. The validate floating-buttons render both verdicts at once.
app_playback open play=True → 200 app_playback close play=False → 409 …repeating, same second
Effect: the "turn on when I open vs when I play" symptom, and the impossible simultaneous 200+500.
Bug C — Spotify ≠ music mode

No server path from phone Spotify → music

spotify is not in PHONE_DISTRACTION_APPS, and the macro only posts Application Closed (YouTube) + the raw trigger string "Spotify: Playback Started", which parses to unknown → handle_phone_activity returns "not_tracked". The spotify_active icon only lights from desktop_mode=="music" (desktop detector), never from the phone.
phone/event body: Application Closed (YouTube) phone/event body: {trigger_that_fired} // raw, ignored
Effect: Spotify on the phone clears YT and does nothing else. Music mode is unreachable from the phone.
Bug D — games have no belt

Slay the Spire enforced while closed

The Telemetry macro drives all games purely off ApplicationLaunched/Closed triggers — which fire on transient surfaces, not true foreground. YouTube got a playback-edge + re-assert belt to fix exactly this; games never did.
Phone activity: app=slay the spire action=open → ENFORCE zap=50 …action=close …action=open …action=close (flapping)
Effect: a spurious "launched" fires the cascade before the "closed" arrives → a zap for an app you never foregrounded.

3 · The two correction mechanisms (and why they don't save you)

MechanismWhat it doesWhy it fails today
Recursive retry
YT_POLL + /yt_poll loopback		 Re-validates with assert={lv=request[assert]}, disables triggers, re-posts a corrected edge. This is the "flash 500 then 200" you see — and it genuinely works when handed a real value. It faithfully re-sends whatever assert it was given. When the upstream value is the garbage from Bug A, the retry just re-sends garbage. It compensates for races (Bug B) but can't invent a correct value.
10-min re-assert belt
RegularIntervalTrigger 600s		 Periodically re-state the current playback boolean so a missed edge self-corrects. Keyed by {trigger_that_fired} instead of a real "is-playing" variable → always misses the dict → always sends the unexpanded literal. The belt is structurally broken (Bug A).

4 · Recommended redesign

Wave 1 (this pass): ship the passthrough / backfill assert — the phone sends one idempotent ?app=…&assert=<bool>&backfill=1 and the server makes its state match. That alone removes the recursion macro, the 200↔409↔422 button churn, and most of the stuck-state problem. Everything else below is documented for follow-up waves. The current recursion/belt version is being archived as a tech demo before replacement.

Answer to "make it generic vs make the server write-through": do both, split by responsibility. Make the server authoritative so the phone doesn't need a retry loop, and generalise the (now-thin) belt to all media/games.

Server: write-through

One idempotent assert, server makes it true

The validate endpoint already has _state_backfill_app_assertion(). Have the phone call /api/state/validate?app=youtube&assert=<bool>&backfill=1. On mismatch the server writes the state to match and returns 200. This collapses the whole YT_POLL recursion into a single call — no client retry, no "flash 500".
Phone: real boolean

Maintain one yt_playing var, not a trigger-name dict

MusicPlayingTrigger start → set yt_playing=true; stop → false. The interval belt and any retry send {lv=yt_playing} — a real value that survives any trigger. Kills Bug A.
Phone: one source of truth per app

Playback edge owns YT; drop the launch edge

Delete the play:False-on-launch behaviour from YT. Foreground-open should not assert playback. Removes the YT-vs-YT_BG race (Bug B).
Generalise the belt

Games re-assert + debounce, same as YT

Route games through a periodic backfill re-assert with a short debounce so a lone "launched" without a sustained foreground doesn't fire the cascade. Server can also require N seconds of sustained open before enforcing. Kills Bug D.
Spotify → music

Add a music-mode entry path

Map spotify playback-start to music (either add "spotify" as a non-enforced media app that sets desktop_mode=music/current_app=spotify, or a dedicated ?mode=music assert). Spotify-stop clears it. Fixes Bug C and lights the ♪ icon from the phone.
Keep, slimmed

Belt stays — suspenders retire

Keep the periodic re-assert (cheap insurance against missed edges) but retire the phone-local recursion macro once the server is write-through. The retry isn't pointless today only because it papers over Bugs A/B — fix those and it's dead weight.
